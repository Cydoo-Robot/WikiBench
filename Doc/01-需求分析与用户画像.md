# 01 · 需求分析与用户画像

> **本文档回答什么问题**
> - 谁是 WikiBench 的用户？他们在什么场景下使用？
> - 每类用户的核心痛点是什么？我们解决哪些、不解决哪些？
> - 功能需求从痛点如何推导？

---

## 1. 用户画像

我们将用户分为三个主要群体，每个群体有不同的使用动机与评价标准。

### 画像 A · 开源实现作者（Builder）

- **典型代表**：ussumant、Ar9av、AgriciDaniel、claude-obsidian 的作者
- **目标**：证明自己的实现比别人好 / 不差
- **现状痛点**：
  - 只能自测自夸，缺乏"别人相信的数据"
  - refactor 后质量变化无法量化 → 怕改坏
  - 想对比其它实现，但没有统一坐标系
- **核心诉求**：
  1. 一条命令跑完全部标准测试
  2. 结果可分享、可引用（带版本号、可复现）
  3. 能看到自己在哪个指标上不如别人

### 画像 B · 研究员 / 学生（Researcher）

- **典型代表**：LLM 记忆 / 知识编辑 / 长上下文方向的研究者
- **目标**：研究 LLM Wiki 范式的基础性质、发论文
- **现状痛点**：
  - 没有公共数据集，每篇论文自己造数据 → 不可比较
  - 单一指标（比如 ROUGE）不足以支撑论文结论
  - 缺少 ground truth 可控的 synthetic corpus
- **核心诉求**：
  1. 有 ground truth 的 synthetic corpus，参数可调
  2. 细粒度指标，能拆解分析
  3. 可扩展：自己定义新任务 / 新指标

### 画像 C · 企业技术决策者（Adopter）⚠️ Phase 2 主战场

- **典型代表**：平台团队负责人、AI 工程 lead、CTO
- **目标**：决定要不要把 LLM Wiki 上生产、上哪一家
- **现状痛点**：
  - Demo 都很漂亮，生产不敢用：怕幻觉、怕丢信息
  - 不知道在自己的数据规模、领域上表现如何
  - 不知道半年后"知识腐烂"会到什么程度
  - 缺少 CI/CD 集成：每次 wiki 更新后要不要上线？
- **核心诉求**：
  1. 在自己语料上一键出健康度报告
  2. 接入 CI，质量下降自动告警
  3. 在多个候选实现之间做**基于自己数据**的横向评测

## 2. 核心使用场景

### 场景 S1 · Builder 做一次回归测试

```
$ wikibench run --impl ussumant --corpus small
Running on corpus=small (50 docs, 100 QA pairs, 10 contradictions)
[ingest]  42.3s  · 182k tokens
[eval]    18.7s  · 3 tasks

 Retrieval Accuracy@1      0.82
 Retrieval Accuracy@3      0.94
 Fidelity Score            0.88
 Hallucination Rate        0.03
 Contradiction Recall      0.60
 Token Ratio               0.12
 Composite Score           0.79   (prev 0.77, +0.02)

 Report: ./runs/ussumant-20260417-1523/report.html
```

### 场景 S2 · Researcher 生成定制 synthetic corpus

```python
from wikibench.corpora import SyntheticGenerator

gen = SyntheticGenerator(
    domain="clinical trial protocols",
    n_documents=200,
    n_facts=500,
    n_contradictions=20,
    contradiction_types=["numeric", "temporal", "entity"],
    seed=42,
)
corpus = gen.build()
corpus.save("./my-corpus")
```

### 场景 S3 · Adopter 做候选方案横向对比（Phase 2）

```
$ WikiBench compare \
    --corpus ./our-internal-docs \
    --impls ussumant,ar9av,claude-obsidian \
    --models gpt-4o-mini,claude-3-5-sonnet \
    --output ./report.html
```

### 场景 S4 · Adopter 接入 CI/CD（Phase 2）

```yaml
# .github/workflows/wiki-quality.yml
- name: WikiBench Quality Gate
  uses: wikibench/WikiBench-action@v1
  with:
    corpus: ./docs
    baseline: main
    fail_on:
      fidelity_drop: 0.05
      hallucination_rise: 0.02
```

## 4. 非功能需求

| 维度 | 要求 | 说明 |
|------|------|------|
| **可复现性** | 同一 corpus+impl+seed 结果一致 | 记录模型版本、随机种子、实现 commit hash |
| **可扩展性** | 新增任务 / 指标 < 50 行代码 | Plugin 式注册 |
| **可移植性** | Windows / macOS / Linux 均可跑 | Python 3.11+ |
| **低接入成本** | 现有实现接入 < 1 小时 | Adapter 只需实现 2 个方法 |
| **成本透明** | 每次评测明确报告 token / $ 成本 | 默认离线语料，不强制调云 API |
| **可验证性** | 所有指标都附带计算方式说明 | 文档化 + 单元测试 |

## 5. 明确**不**做的事

为避免 scope creep，以下明确不在项目范围内：

- ❌ **做自己的 LLM Wiki 实现** —— 我们是裁判，不是选手
- ❌ **做 RAG 评测** —— 已有 RAGAS 等成熟工具
- ❌ **托管式 SaaS 服务**（至少 Phase 1 不做）—— 保持开源中立
- ❌ **通用 LLM 评测框架** —— 聚焦 *知识编译* 场景
- ❌ **评测编译**中**的过程**（如中间 chain-of-thought）—— 只看输入（corpus）和输出（wiki）
