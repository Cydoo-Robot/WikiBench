## 1. 架构总览

### 1.1 三层架构

```
┌────────────────────────────────────────────────────────────────┐
│                      Layer 3 · Metrics                         │
│     效率指标  ·  质量指标  ·  可维护性指标  ·  综合评分       │
├────────────────────────────────────────────────────────────────┤
│                      Layer 2 · Tasks                           │
│   Retrieval  ·  Fidelity  ·  Contradiction  ·  Staleness ···   │
├────────────────────────────────────────────────────────────────┤
│                      Layer 1 · Corpora                         │
│      Synthetic  ·  Small  ·  Medium  ·  Large (real-world)     │
└────────────────────────────────────────────────────────────────┘
                              ▲
                              │ 统一评测接口
                              │
┌─────────────────────────────┴──────────────────────────────────┐
│                    Layer 0 · Adapters                          │
│   ussumant · Ar9av · AgriciDaniel · claude-obsidian · custom  │
└────────────────────────────────────────────────────────────────┘
```

- **Layer 0（适配器）**：被测对象接入层，把各种 LLM Wiki 实现包装成统一接口
- **Layer 1（语料）**：提供标准化输入数据，包括可控的 synthetic 和真实世界 corpus
- **Layer 2（任务）**：定义"问知识库什么问题"，产出可验证的答案-评分对
- **Layer 3（指标）**：把任务结果聚合成可比较的分数

## 2. 核心数据流

### 2.1 端到端流程（一次 benchmark run）

```
1. CLI 接收 --impl --corpus --tasks 参数
        │
        ▼
2. Runner 加载 Corpus（Layer 1）
        │  产出：docs[], qa_pairs[], ground_truth
        ▼
3. Runner 实例化 WikiAdapter（Layer 0）
        │
        ▼
4. Adapter.ingest(docs)  ◀── 被测实现把 docs 编译成 wiki
        │  记录：ingest_tokens, ingest_latency
        ▼
5. 对每个 Task：
     5a. task.prepare(corpus)        → queries[]
     5b. for q in queries:
            response = adapter.query(q)   ◀── 记录 query_tokens
            score = task.score(q, response)
     5c. aggregate task scores
        │
        ▼
6. MetricEngine 合并所有任务产出 → final metrics
        │
        ▼
7. ResultStore 持久化
        │
        ▼
8. Reporter 生成 HTML / JSON / Markdown
```

### 2.2 关键数据实体

```python
# 伪代码级别的核心数据模型（详见 05-评测接口与适配器规范）

Corpus:
    id: str
    docs: list[Document]
    metadata: CorpusMetadata       # 领域、规模、synthetic?、seed
    ground_truth: GroundTruth      # qa_pairs, fidelity_claims, contradictions

Document:
    id: str
    path: str                      # 模拟文件路径，保留目录结构语义
    content: str
    modality: Literal["markdown", "transcript", "chat", "email", ...]
    timestamp: datetime | None     # 用于 staleness 任务

WikiAdapter:  # 第三方实现需要继承
    def ingest(self, docs: list[Document]) -> IngestResult
    def query(self, q: str) -> QueryResponse
    # 可选：
    def update(self, new_docs, removed_docs) -> IngestResult  # 增量
    def export_wiki(self) -> str                               # 调试用

Task:
    id: str
    def prepare(self, corpus: Corpus) -> list[Query]
    def score(self, query: Query, response: QueryResponse) -> Score

BenchmarkResult:
    run_id: str
    impl: str
    corpus_id: str
    timestamp: datetime
    ingest: IngestStats
    per_task_scores: dict[str, Score]
    final_metrics: dict[str, float]
    environment: RunEnvironment    # python/model 版本、commit hash 等
```

## 3. 组件详细职责

### 3.1 Runner（编排器）

- **职责**：读取配置、初始化各组件、驱动执行流程、记录日志
- **不做**：具体的任务打分逻辑、indexing 策略
- **关键设计**：
  - 支持 **dry-run**（只列任务不执行）
  - 支持 **resume**（从上次中断处继续，长 corpus 可能跑数小时）
  - 支持 **并行任务**（tasks 之间无依赖可并行）

### 3.2 Corpus Loader

- **职责**：加载标准语料 / 用户自有语料，校验完整性
- **格式约定**：
  - 根目录有 `manifest.yaml`（元数据 + ground_truth 索引）
  - `docs/` 子目录存放文档
  - `ground_truth/` 子目录存放 QA、claims、contradictions
- **校验**：`manifest.yaml` 必须通过 pydantic schema；hash 校验可选

### 3.3 Wiki Adapter（关键扩展点）

- **职责**：把第三方实现包装成标准接口
- **设计原则**：
  - *Thin is better than thick*：adapter 不做逻辑，只做协议转换
  - *Timeout & resource guard*：adapter 必须声明最大 token / 最大时长
  - *成本追踪*：所有 LLM 调用统一经过 `TokenCounter` 中间件
- **内置 adapter**（MVP）：
  - `NaiveAdapter` —— 基线：把全部 docs 塞上下文
  - `SimpleSummaryAdapter` —— 基线：LLM 摘要后存储
  - `ReferenceWikiAdapter` —— 参考实现（类 Karpathy 原型）
- **外部 adapter**：通过 entry_points / 插件机制发现

### 3.4 Task Suite

- **职责**：产出 queries，评分 responses
- **设计原则**：
  - 每个 task 是一个独立模块，可单独启用/禁用
  - task 必须支持"无 LLM 打分"（尽量用规则 / exact match / F1），实在需要 LLM-as-judge 时必须可替换模型
- **MVP 任务**：详见 `04-评测任务与指标体系.md`

### 3.5 Metric Engine

- **职责**：把各 task 的原始分数聚合成可比较的指标
- **关键设计**：
  - 区分 *raw metric*（任务直接产出）和 *aggregated metric*（跨任务合成）
  - *composite score* 算法**公开、可重现**，不做黑盒加权
  - 所有 aggregation 算法都在 `metrics/aggregators/` 下有对应文件

### 3.6 Result Store

- **职责**：持久化 run 结果
- **选型**：
  - 默认：SQLite + JSON blob（本地，零依赖）
  - 扩展：Postgres（团队共享） / 云存储（leaderboard 后端）
- **Schema**：见 `06-技术栈与工程规范.md`

### 3.7 Reporter

- **职责**：把 BenchmarkResult 渲染成人类可读形式
- **输出格式**：
  - `console` —— Rich 彩色表格
  - `json` —— 机器可读
  - `html` —— 独立页面，可分享
  - `markdown` —— 贴到 PR / issue
- **对比模式**：支持 `report --compare run_a run_b`

## 4. 可插拔扩展点

| 扩展点 | 机制 | 典型用户 |
|--------|------|---------|
| Wiki 实现 | `WikiAdapter` 子类 + entry_point | Builder |
| 语料库 | 放到 `corpora/` 或发布 pip 包 | Researcher / 企业 |
| 评测任务 | `Task` 子类 + 注册 | Researcher |
| 指标 | `Metric` 函数 + 注册 | Researcher |
| LLM-as-judge 后端 | `JudgeModel` 接口 | 所有 |
| 报告格式 | `Reporter` 插件 | 企业 / CI |

> **插件发现机制**：Python `entry_points`（`pyproject.toml` 中 `[project.entry-points."wikibench.adapters"]`），这样第三方只需 `pip install wikibench-adapter-ussumant` 即可被主程序识别。

## 5. 架构原则

1. **正交性**：Corpus、Task、Metric、Adapter 四者之间相互独立，通过数据契约耦合
2. **可复现性优先**：任何随机性都需要 seed；任何 LLM 调用都需要模型版本记录
3. **离线优先**：默认只跑本地 corpus，不强制联网；LLM-as-judge 可 opt-in
4. **成本透明**：所有 token / $ 成本全程可见，报告里明示
5. **对贡献者友好**：接入门槛 < 1 小时；新增任务 / 指标不需要改核心
6. **版本化一切**：corpus、task、metric 都有 version；报告里记录所有版本

## 6. 非目标 & 反模式

- ❌ **不做端到端 autonomous agent**：WikiBench 是评测工具，不是智能 pipeline
- ❌ **不隐式调用 LLM**：所有 LLM 调用必须显式配置，不静默消费用户的 API key
- ❌ **不做实时监控 dashboard**（Phase 1）：保持 batch 评测语义
- ❌ **不引入重型依赖**：核心包不依赖 LangChain 全家桶；必要时在子模块里隔离

## 7. 未决议题

- [ ] **异步 vs 同步**：LLM adapter 是否需要原生 async？当前倾向：同步为主 + 内部并发，avoid 传染性 async
- [ ] **缓存层**：是否内置 adapter response 缓存？（同一 corpus + query 可复用）
  - 倾向：内置，基于内容 hash，默认开启但可关闭
- [ ] **分布式执行**：large corpus 是否需要分布式 task runner？
  - 倾向：Phase 2 考虑，MVP 用多进程足够
