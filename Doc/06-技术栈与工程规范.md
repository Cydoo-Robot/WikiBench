## 1. 技术栈总览

| 层 | 选型 | 理由 |
|----|------|------|
| 语言 | **Python 3.11+** | AI 生态默认、LLM SDK 最全、贡献者门槛低 |
| 包管理 | **uv** | 快；lockfile 强；兼容 pip |
| 数据模型 | **pydantic v2** | 运行时校验 + schema 生成 |
| CLI | **Typer + Rich** | 声明式、输出好看 |
| 配置 | **pydantic-settings + YAML** | 嵌套结构友好 |
| 日志 | **structlog** | JSON 结构化日志，利于 CI 解析 |
| 工作流编排 | **自研轻量 runner** + 可选 LangGraph | 避免核心包强耦合重型依赖 |
| LLM 抽象 | **litellm**（统一 API） | 一次接入 OpenAI / Anthropic / 本地 / Azure |
| Token 计数 | **tiktoken + 模型各自 tokenizer** | 精确计费 |
| 缓存 | **diskcache** | 本地零依赖 |
| 结果存储 | **SQLite（默认）+ JSON blob** | 零运维；Postgres 作为可选扩展 |
| 报告 | **Jinja2 + 内嵌 HTML** | 单文件可分享 |
| 测试 | **pytest + hypothesis + pytest-asyncio** | 行业标准 |
| 代码质量 | **ruff（lint+format）+ mypy --strict**（核心模块） | 严格但快 |
| 前端 / Leaderboard | **Next.js + Tailwind + shadcn/ui** | Phase 1 后期交付 |
| 文档 | **MkDocs Material** | 好看、Markdown 直接构建 |
| CI | **GitHub Actions** | 默认生态 |

## 3. 关键设计约定

### 3.1 分层依赖规则

```
cli     → runner, reporters, storage
runner  → adapters, tasks, metrics, runtime, corpora
tasks   → models, runtime, judge
metrics → models
adapters→ models, runtime        # 不依赖 tasks / metrics
corpora → models, runtime
runtime → models
models  → (纯 pydantic, 零业务依赖)
```

禁止反向依赖，用 `ruff`/`import-linter` 强制校验。

### 3.2 公共 API 暴露

```python
# src/wikibench/__init__.py
from wikibench.runner import Runner
from wikibench.adapters import WikiAdapter
from wikibench.tasks import Task
from wikibench.metrics import register_metric
from wikibench.models import Document, Query, QueryResponse, Corpus, BenchmarkResult

__all__ = [
    "Runner", "WikiAdapter", "Task", "register_metric",
    "Document", "Query", "QueryResponse", "Corpus", "BenchmarkResult",
]
```

其它模块视为 internal，改动不保证向后兼容。

### 3.3 错误处理

- 统一异常基类：`WikiBenchError`
- 子类：`AdapterError`, `CorpusError`, `TaskError`, `MetricError`, `ConfigError`
- Runner 捕获 adapter 异常 → 写入 `BenchmarkResult.warnings`，不中断其它任务

## 4. 代码质量门槛

### 4.1 强制 check（CI 必过）

- `ruff check` —— lint
- `ruff format --check` —— 风格
- `mypy --strict src/wikibench/{models,runtime,adapters,tasks,metrics}` —— 核心模块严格
- `pytest tests/unit tests/integration` —— 单元 + 集成测试
- `pytest tests/e2e --corpus synthetic-tiny` —— 端到端跑通（10 docs）

### 4.2 命名规范

- 模块、包：snake_case
- 类：PascalCase
- 常量：UPPER_SNAKE
- 私有前缀：`_`
- Task / Metric id：snake_case、稳定不变
- Adapter name：小写短横线或下划线，在 registry 唯一

### 4.3 文档字符串

- 所有公共类 / 函数必须有 docstring
- 风格：Google style
- 核心抽象类（`WikiAdapter` 等）docstring 必须包含 *Example* 段落

## 5. 开发流程

### 5.1 起步

```bash
git clone <repo>
cd WikiBench
./scripts/bootstrap-dev.sh            # 或 bootstrap-dev.ps1
# ↑ 安装 uv、创建 venv、uv sync、pre-commit install

wikibench --help
wikibench run --impl naive --corpus synthetic-tiny
```

### 5.2 分支策略

- `main`：永远可发布
- `feat/<short-desc>`：功能分支
- `fix/<issue#>`：修复分支

### 5.3 Commit 规范

采用 Conventional Commits：

```
feat(tasks): add contradiction detection task
fix(runner): handle adapter timeout gracefully
docs(corpora): document manifest schema
```

### 5.4 版本与发布

- 版本号在 `src/wikibench/__version__.py` 维护
- 发布由 `scripts/release.py` 驱动，自动：
  - 更新 CHANGELOG.md
  - 打 tag
  - 推送 PyPI
  - 触发 leaderboard 刷新

## 6. 测试策略

### 6.1 金字塔

```
            ┌───────────┐
            │   e2e     │   5%   —— 跑 synthetic-tiny，验证 CLI 到报告全链路
            ├───────────┤
            │integration│  25%   —— Runner + mock adapter；Corpus 加载；Task 打分
            ├───────────┤
            │   unit    │  70%   —— 数据模型、生成器步骤、指标函数
            └───────────┘
```

### 6.2 关键测试约定

- **确定性**：所有用 LLM 的测试 mock 掉；真调用放 `tests/live/` 手动跑
- **Hypothesis**：对 ground_truth / manifest schema 做 property test
- **Golden files**：报告渲染用 snapshot 测试
- **adapter smoke test fixture**：所有 adapter 共用 `tests/adapter_contract/`

## 7. 配置管理

### 7.1 配置文件优先级

```
CLI flag  >  环境变量  >  项目 wikibench.yaml  >  用户 ~/.wikibench/config.yaml  >  默认值
```

### 7.2 `wikibench.yaml` 示例

```yaml
llm:
  default_model: gpt-4o-mini
  providers:
    openai:
      api_key: ${OPENAI_API_KEY}
    anthropic:
      api_key: ${ANTHROPIC_API_KEY}

cache:
  enabled: true
  dir: ./.wikibench-cache

runner:
  max_concurrent_tasks: 4
  default_timeout_s:
    query: 60
    ingest: 1800

judge:
  model: gpt-4o
  disagreement_policy: conservative    # conservative | fallback_human

cost:
  hard_limit_usd: 20.0
  warn_at_usd: 10.0
```

### 7.3 敏感数据

- 密钥一律走环境变量 / `.env`（gitignore）
- 报告中的 `environment` 字段自动脱敏

## 8. 性能 / 资源预算

| 场景 | 文档数量 | 预期时长 | token 成本（gpt-4o-mini） |
|------|---------|---------|------------------------|
| `synthetic-tiny` | 10 docs | < 30s | < $0.02 |
| `small` | **500+ docs** | 15–30 min | $1–3 |
| `medium` | **2000+ docs** | 1–3 h | $5–15 |
| `large` | **50000+ docs** | 10–48 h | $100–400 |

> **Large corpus 说明**：50000 docs 级别建议拆分批次运行，Runner 支持 resume；成本估算基于 ingest（编译阶段），评测阶段（query × task 数）另计。建议 large 只在需要时按需运行，不作为 CI 默认。

Runner 必须在启动时打印预算估算，并在 `hard_limit_usd` 前强制确认。
