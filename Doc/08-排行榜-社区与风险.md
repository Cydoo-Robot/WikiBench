## 1. Leaderboard 设计

### 1.1 产品形态

```
 ┌───────────────────────────────────────────────────────────────┐
 │  WikiBench Leaderboard                       [v0.5.0]         │
 │                                                               │
 │  Corpus: [synthetic-001-saas ▾]  Model: [gpt-4o-mini ▾]       │
 │  Tier:   [ small | medium | large | adversarial ]             │
 │                                                               │
 │  Rank Impl              Quality  Efficiency  Maintain  ∑      │
 │  ──── ────────────────  ───────  ──────────  ────────  ─────  │
 │   1   ussumant@0.4.1     0.82     0.74        0.68    0.76    │
 │   2   reference-wiki     0.79     0.71        0.72    0.74    │
 │   3   ar9av@0.2.3        0.85     0.58        0.65    0.71    │
 │   4   claude-obsidian    0.73     0.88        0.55    0.70    │
 │   …                                                           │
 └───────────────────────────────────────────────────────────────┘
```

### 1.2 核心原则

- **多榜并存**：不设"唯一分数排行"。按维度、按 corpus、按模型独立排序
- **版本透明**：每条记录绑定 `impl@version`、`corpus@version`、`wikibench@version`
- **可审计**：每个分数可点开查看 run detail（环境、配置、成本）
- **分层**：按 `tier`（synthetic / small / medium / large / adversarial）分开排
- **不做跨 tier 统一排名**（避免"在简单数据上刷分"）

### 1.3 提交流程

**开源作者自助提交**：
```
 作者本地                        WikiBench CI
   │                                │
   │ wikibench run ... ──────────▶  │
   │                                │  跑相同 adapter 版本
   │                                │  跑相同 corpus 版本
   │ 提交 PR 到 leaderboard repo    │  跑相同 seed
   │    ├─ result.json              │
   │    └─ adapter manifest         │  结果一致则入库
   │                                │  不一致则 reject + log
   │◀──────────────── 公开可见     │
```

- 所有提交走 PR，双重跑分（本地 + CI 各跑一次）
- 两次结果差 > 5% 视为不可复现，reject
- 入榜需 adapter 通过 `verify-adapter` 契约测试

### 1.4 反 gaming 机制

| 风险 | 缓解 |
|------|------|
| 在 synthetic corpus 上 overfit | 每季度更换 hold-out 对抗性 corpus，不公开 seed |
| adapter 在 ingest 时泄题（读 ground truth 文件） | 强制目录隔离，runner 只把 docs/ 传给 adapter |
| adapter 内部调更贵模型刷分 | 模型版本强制记录，按成本分桶排序 |
| 选择性报告 | 必须同时报告全部指标，不允许只发一项 |
| 用 LLM-as-judge 的模型做 adapter backend | 框架侧拒绝，runner 检测并 reject |

### 1.5 技术栈

- 前端：Next.js 14 + Tailwind + shadcn/ui
- 数据：GitHub 作为**存储真源**（`leaderboard-data` repo 的 JSON 文件）
- 构建：Cloudflare Pages 自动部署
- 搜索：客户端全量（数据量 < 10k 条时足够）

## 2. 社区运营

### 2.1 启动策略（MVP 发布前后）

- **Pre-launch（Phase 0 末）**
  - 公开 repo + Doc（就是本目录）
  - 在 Karpathy 原推评论区留一条**低调说明**（不引战）
  - 找 3–5 个 LLM Wiki 作者**私信**征求设计意见
  - 开 Discord / GitHub Discussions

- **Launch（MVP 发布当天）**
  - 发 X/Twitter thread：一页截图 + 三张核心图 + repo 链接
  - 投递 Hacker News（Show HN）
  - 给几个 AI newsletter 写 submission（Import AI / Interconnects / The Sequence）
  - 中文圈：机器之心 / 量子位 / 少数派

- **Post-launch**
  - 每周 dev log（blog + Discord）
  - 每月录制一次 5 分钟短视频 walkthrough
  - 季度性"评测报告"（对公开 adapter 做跨版本对比）

### 2.2 外部接入激励

- **"Adapter 接入挑战"**：接入的 LLM Wiki 实现作者列入 README Credits
- **RFC 流程**（Phase 1.5 后）：破坏性 API 改动需发 RFC + 14 天公示期，给已接入的社区 adapter 适配时间
- **Fork-friendly**：Apache 2.0 + 明确声明"本项目是基础设施，欢迎 fork"

## 3. 风险登记簿

### 3.1 战略风险

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 竞品（学术团队 / 大公司）先发布类似基准 | 中 | 高 | 加快 MVP；差异化聚焦工程可用性 |
| Karpathy 或其他大 V 背书另一个基准 | 低-中 | 高 | 保持设计开放性，主动联系领袖对齐 |
| LLM Wiki 范式本身被证伪 / 被新范式替代 | 低 | 毁灭性 | 早期接受；指标定义足够通用可迁移 |
| 窗口期关闭（超过 6 个月还没立标） | 中 | 高 | 严控 MVP 范围，不做 nice-to-have |

### 3.2 技术风险

| 风险 | 缓解 |
|------|------|
| Synthetic corpus 有系统性偏差被社区质疑 | 公开生成 pipeline + 抽样人工审查 |
| LLM-as-judge 引入模型偏差 | 双判 + 规则优先 + 开放 judge 替换 |
| 不同实现的接口难以抽象 | Adapter 做薄，允许 adapter 声明能力 flag |
| token 计数在不同 tokenizer 下不可比 | 统一以 tokens 为主，附 char count 兜底 |
| 评测结果不稳定（随机性） | 强制 seed + 多次重跑取均值（选项） |

### 3.3 社区风险

| 风险 | 缓解 |
|------|------|
| Adapter 作者不愿接入（"被评判"感受差） | 先只接入愿意的；positive framing（"你的实现已被收录"） |
| 评分结果引发开源社区纷争 | 文档强调"不做唯一分数排名"；多维度呈现 |
| 企业不愿把自己数据用于 synthetic 参考 | 用公开数据起步；后期单独 DPA |

### 3.4 运营风险

| 风险 | 缓解 |
|------|------|
| leaderboard 托管成本失控 | 静态站点优先；有 CDN 缓存；无后端 |
| 用户误跑大 corpus 烧 API 费用 | 默认 `cost.hard_limit_usd` 20$，跨线须显式确认 |
| 报告安全问题（用户私密语料被上传） | 明确"不收集任何数据"；所有遥测 opt-in |

## 4. 许可与知识产权

- **核心代码**：Apache 2.0
- **Synthetic corpus**：CC-BY-4.0
- **真实公开 corpus**：遵循原始 License，标注 attribution
- **Adapter 包**：作者自选，建议 Apache 2.0 / MIT
- **Leaderboard 数据**：CC-BY-4.0，允许研究引用

**商标**：`WikiBench` / `WikiProbe` 考虑注册（Phase 2），防止被企业抢注。

## 5. 遥测与隐私

**原则：零强制遥测。**

- 默认**不上传任何数据**（语料、结果、环境）
- 可选 `wikibench telemetry enable` 后：
  - 只上传：wikibench 版本、运行成功/失败、任务总数
  - 从不上传：语料内容、query 内容、adapter 输出
  - 公开遥测代码路径，用户可审计

## 6. 未决议题汇总

整合自 00–07 各文档末尾的未决议题，按优先级排序：

### P0（Phase 0 内必决）
- [ ] 命名：WikiBench 单一 / WikiBench + WikiProbe 双品牌
- [ ] License：Apache 2.0（建议） vs MIT
- [ ] 第一个外部 adapter 合作对象（ussumant / Ar9av / ...）
- [ ] 中/英文 README 主副关系
- [ ] Launch 时机策略

### P1（Phase 1 内决）
- [ ] LLM-as-judge 的 disagreement 策略（conservative / fallback_human）
- [ ] Retrieval answer_type 是否含 "reasoning"
- [ ] Composite Score 跨 tier 归一化方案
- [ ] Adapter 是否允许 query 时动态查原始 docs
- [ ] MVP 目标窗口 6 周 / 8 周
- [ ] export_wiki 是否作为硬性要求

### P2（Phase 1.5 内决）
- [ ] Medium/Large corpus 标注方式（LLM 预标注 + 人工抽查）
- [ ] 中/英双语 corpus 对齐方式
- [ ] 对抗性 corpus 的构造手法（生成式 / 人工 / 混合）
- [ ] Adapter 异步 API 支持策略
- [ ] 对抗性 corpus 是否公开 seed

### P3（Phase 2+）
- [ ] WikiProbe SaaS 产品化路径
- [ ] 核心包是否拆 monorepo
- [ ] 云端 benchmark runner 是否提供
- [ ] 商标注册与品牌资产策略

## 8. 结语

**这不是一个"如果成功就更好"的项目，而是一个窗口期明确、标准位置空缺的基础设施机会。**

- 第一个能跑的、文档齐全的、设计不崩的基准库，会自然地成为生态默认值。
- 不需要做到完美，但需要做得**可信、可复现、可扩展**。
- 真正的胜负手是：**能否让 5 个以上的主流 LLM Wiki 实现官方接入并继续用它做回归测试**。做到这一步，标准就成立了。

剩下的事，和 SWE-bench 从发布到被 Anthropic / OpenAI 原生支持那 6 个月差不多——**执行，公开，迭代**。
