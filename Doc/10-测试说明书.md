# WikiBench 测试说明书

> **本文档回答什么问题**：如何在本仓库中运行测试、测试如何分层、哪些用例依赖外部资源或 LLM、以及 CI 与本地开发时的常用命令。

---

## 1. 环境与前置条件

| 项目 | 要求 |
|------|------|
| Python | `>= 3.11`（与 `pyproject.toml` 一致） |
| 依赖管理 | **uv**（见 [06-技术栈与工程规范](./06-技术栈与工程规范.md)） |
| 安装 | 仓库根目录执行 `uv sync --extra dev`（测试工具在 `[project.optional-dependencies]` → `dev`） |

若已安装过含 `dev` 的环境，可直接使用下文中的 `uv run pytest`。

命令行端到端 walkthrough（与测试互补）见仓库根目录 [`examples/README.md`](../examples/README.md)。

## 2. 常用命令

```bash
# 运行全部测试（默认配置见 pyproject.toml [tool.pytest.ini_options]）
uv run pytest

# 指定目录或文件
uv run pytest tests/unit/
uv run pytest tests/integration/
uv run pytest tests/e2e/test_full_run.py -v
uv run pytest tests/adapter_contract/ -v

# 与契约测试等价的手动命令（默认 mock，不耗 API）
uv run wikibench verify --adapter naive --corpus corpora/synthetic/tiny

# 只跑某一类 marker（见下文）
uv run pytest -m unit
uv run pytest -m integration
uv run pytest -m e2e

# 排除慢速用例（若未来使用 @pytest.mark.slow）
uv run pytest -m "not slow"

# 覆盖率（可选用；未安装时先：uv add --dev pytest-cov）
uv run pytest --cov=wikibench --cov-report=term-missing
```

默认 `addopts` 为 `-ra -q --tb=short`：失败时简短回溯、结束时列出摘要。

## 3. 测试目录与分层

| 目录 | 约定 | 典型内容 |
|------|------|----------|
| `tests/unit/` | 快速、无外部 I/O 或仅内存/临时目录 | 模型、工具函数、缓存、Reporter、SyntheticGenerator 等 |
| `tests/integration/` | 读仓库内语料、跑 Runner/Task，**不依赖真实 LLM API** | `corpora/synthetic/tiny` 加载、Runner、任务打分管线 |
| `tests/e2e/` | 通过 Typer `CliRunner` 调用 `wikibench` CLI，**端到端** | `wikibench run` → 结果落盘 + SQLite + `wikibench report` |
| `tests/adapter_contract/` | 内置 + 社区适配器契约 | 内置：`naive` / `simple_summary` / `reference_wiki`；社区：见 **§3.1** |

**仓库内固定语料**：`corpora/synthetic/tiny`（hand-crafted synthetic-tiny）。`tests/conftest.py` 中 `tiny_corpus_path` fixture 会断言该路径存在；缺失时相关测试失败。

### 3.1 社区 Adapter 评测沙箱（llm-wiki-compiler / obsidian-wiki）

与 [07-开发路线图与MVP计划 §4.1.1](./07-开发路线图与MVP计划.md) **第三方评测沙箱原则**一致：WikiBench **不在仓库内自造**与上游等价的编译/查询逻辑；第三方实现放在 **隔离工作目录**（clone 官方仓库、固定 tag/commit），Adapter 只做 **薄封装**（准备语料 → 调真实 CLI 或约定入口 → 解析产物）。密钥与路径走 **环境变量**，不写死进仓库。

| 项目 | 规划要点 |
|------|----------|
| **沙箱根目录** | 默认 **`.wikibench-sandboxes/`**（仓库根下、点目录通常不进 Git）；可用环境变量 **`WIKIBENCH_SANDBOX_ROOT`** 改到本机任意路径。代码中建议子目录名：`llm-wiki-compiler`、`obsidian-wiki`（见 `wikibench.adapters.community.sandbox`）。 |
| **A · llm-wiki-compiler**（atomicmemory） | **Node + `llmwiki` CLI**。沙箱内 `git clone` 上游或按 README 全局安装 `llm-wiki-compiler`；评测时 **`subprocess` 调用 `llmwiki ingest` / `compile` / `query`**，读取 `wiki/` 等产物。依赖 **真实 LLM**，**不可**与 **`WIKIBENCH_LLM_MOCK=1`** 同时使用（Adapter 会拒绝）。 |
| **B · obsidian-wiki**（Ar9av） | **clone 上游** [Ar9av/obsidian-wiki](https://github.com/Ar9av/obsidian-wiki)，遵循 `SETUP.md`、`.skills/` 与 vault 约定；若无完整无头 CLI，允许 **semi-automated**：通过 WikiBench **`llm_call`** 加载沙箱内 **`.skills/**/*.md`** 作为 prompt 一部分（与「使用官方技能」一致，非平行自研提示词）。配置项 **`obsidian_wiki_repo`** 指向该 clone（或布局一致的目录）。**可与 mock LLM 跑通契约**（CI 使用 `tests/fixtures/obsidian_wiki_minimal`）。 |

**测试落点**：

- **`ObsidianWikiAdapter`**：`tests/adapter_contract/test_community_obsidian.py` 使用仓库内 **`tests/fixtures/obsidian_wiki_minimal/.skills/`** + `run_adapter_verify`（默认 **mock**），CI 必跑。
- **`LLMWikiCompilerAdapter`**：`tests/adapter_contract/test_community_llmwiki.py` **默认 skip**；本机已安装 **`llmwiki`** 且具备 API 时，设置 **`WIKIBENCH_RUN_LLMWIKI_INTEGRATION=1`**，并**不要**设置 mock，再运行该文件（主 CI 不装 Node，与 [07 §9.1](./07-开发路线图与MVP计划.md) 可选 job 一致）。

**手动 smoke**（与契约测试等价思路）：

```bash
# Obsidian 编排器 + 官方布局的 obsidian_wiki_repo（示例路径请改为本机 clone）
uv run wikibench verify -a obsidian_wiki --corpus corpora/synthetic/tiny \
  --config /path/to/adapter_config.yaml   # YAML 内含 obsidian_wiki_repo: "D:/sandboxes/obsidian-wiki"

# llm-wiki：需真实 API，勿设 WIKIBENCH_LLM_MOCK
uv run wikibench verify -a llm_wiki_compiler --corpus corpora/synthetic/tiny --no-mock
```

## 4. Pytest Markers

在 `pyproject.toml` 的 `[tool.pytest.ini_options]` → `markers` 中注册：

| Marker | 含义 |
|--------|------|
| `unit` | 快速、无 I/O |
| `integration` | 可做本地 I/O，无 LLM 真调用 |
| `e2e` | 完整 CLI 管线；当前实现使用 **mock LLM**，**不访问网络** |
| `slow` | 耗时 > 30s（预留） |

说明：部分文件仍用目录约定区分层级，**并非所有测试类都显式打了上述 marker**；需要筛选时可用路径（如 `tests/e2e/`）或逐步补全 marker。

## 5. LLM 与网络

- **统一入口**：`wikibench.runtime.llm.llm_call`。若设置环境变量 **`WIKIBENCH_LLM_MOCK=1`** 或 **`WIKIBENCH_TEST_MODE=1`**（值为 `1` / `true` / `yes`），则返回固定桩响应，**不调用真实 API**。
- **单元测试**（如 `tests/unit/test_llm.py`）通过 `autouse` fixture 设置 mock，并把缓存目录指到临时路径，避免污染项目下的 `.wikibench-cache/`。
- **端到端测试**（`tests/e2e/test_full_run.py`）在运行前 `monkeypatch.setenv("WIKIBENCH_LLM_MOCK", "1")`，保证 CI 可重复、无需密钥。

若需对真实模型做手工或夜间回归，应在本地显式**取消** mock 并配置好各 provider 的 API Key；**不要将密钥写入仓库或测试文件**。

## 7. 覆盖率与质量门

- `pyproject.toml` 中 `[tool.coverage.*]`：`source = ["src/wikibench"]`，`fail_under = 70`（需在 CI 中显式调用带 coverage 的 pytest 才会生效）。
- 静态检查：**Ruff**、**mypy**（见 `pyproject.toml`）；与测试互补，建议在提交前一并执行。
