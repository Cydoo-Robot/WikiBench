# WikiBench 测试说明书

> **本文档回答什么问题**：如何在本仓库中运行测试、测试如何分层、哪些用例依赖外部资源或 LLM、以及 CI 与本地开发时的常用命令。

---

## 1. 环境与前置条件

| 项目 | 要求 |
|------|------|
| Python | `>= 3.11`（与 `pyproject.toml` 一致） |
| 依赖管理 | **uv**（见 [06-技术栈与工程规范](./06-技术栈与工程规范.md)） |
| 安装 | 仓库根目录执行 `uv sync --extra dev`（测试工具在 `[project.optional-dependencies]` → `dev`） |

若已安装过含 `dev` 的环境，可直接使用下文中的 `uv run pytest`。

命令行端到端 walkthrough（与测试互补）见仓库根目录 [`examples/README.md`](../examples/README.md)。

## 2. 常用命令

```bash
# 运行全部测试（默认配置见 pyproject.toml [tool.pytest.ini_options]）
uv run pytest

# 指定目录或文件
uv run pytest tests/unit/
uv run pytest tests/integration/
uv run pytest tests/e2e/test_full_run.py -v
uv run pytest tests/adapter_contract/ -v

# 与契约测试等价的手动命令（默认 mock，不耗 API）
uv run wikibench verify --adapter naive --corpus corpora/synthetic/tiny

# 只跑某一类 marker（见下文）
uv run pytest -m unit
uv run pytest -m integration
uv run pytest -m e2e

# 排除慢速用例（若未来使用 @pytest.mark.slow）
uv run pytest -m "not slow"

# 覆盖率（可选用；未安装时先：uv add --dev pytest-cov）
uv run pytest --cov=wikibench --cov-report=term-missing
```

默认 `addopts` 为 `-ra -q --tb=short`：失败时简短回溯、结束时列出摘要。

## 3. 测试目录与分层

| 目录 | 约定 | 典型内容 |
|------|------|----------|
| `tests/unit/` | 快速、无外部 I/O 或仅内存/临时目录 | 模型、工具函数、缓存、Reporter、SyntheticGenerator 等 |
| `tests/integration/` | 读仓库内语料、跑 Runner/Task，**不依赖真实 LLM API** | `corpora/synthetic/tiny` 加载、Runner、任务打分管线 |
| `tests/e2e/` | 通过 Typer `CliRunner` 调用 `wikibench` CLI，**端到端** | `wikibench run` → 结果落盘 + SQLite + `wikibench report` |
| `tests/adapter_contract/` | 内置适配器契约 | 对 `naive` / `simple_summary` 调用 `run_adapter_verify`（mock LLM） |

**仓库内固定语料**：`corpora/synthetic/tiny`（hand-crafted synthetic-tiny）。`tests/conftest.py` 中 `tiny_corpus_path` fixture 会断言该路径存在；缺失时相关测试失败。

## 4. Pytest Markers

在 `pyproject.toml` 的 `[tool.pytest.ini_options]` → `markers` 中注册：

| Marker | 含义 |
|--------|------|
| `unit` | 快速、无 I/O |
| `integration` | 可做本地 I/O，无 LLM 真调用 |
| `e2e` | 完整 CLI 管线；当前实现使用 **mock LLM**，**不访问网络** |
| `slow` | 耗时 > 30s（预留） |

说明：部分文件仍用目录约定区分层级，**并非所有测试类都显式打了上述 marker**；需要筛选时可用路径（如 `tests/e2e/`）或逐步补全 marker。

## 5. LLM 与网络

- **统一入口**：`wikibench.runtime.llm.llm_call`。若设置环境变量 **`WIKIBENCH_LLM_MOCK=1`** 或 **`WIKIBENCH_TEST_MODE=1`**（值为 `1` / `true` / `yes`），则返回固定桩响应，**不调用真实 API**。
- **单元测试**（如 `tests/unit/test_llm.py`）通过 `autouse` fixture 设置 mock，并把缓存目录指到临时路径，避免污染项目下的 `.wikibench-cache/`。
- **端到端测试**（`tests/e2e/test_full_run.py`）在运行前 `monkeypatch.setenv("WIKIBENCH_LLM_MOCK", "1")`，保证 CI 可重复、无需密钥。

若需对真实模型做手工或夜间回归，应在本地显式**取消** mock 并配置好各 provider 的 API Key；**不要将密钥写入仓库或测试文件**。

## 7. 覆盖率与质量门

- `pyproject.toml` 中 `[tool.coverage.*]`：`source = ["src/wikibench"]`，`fail_under = 70`（需在 CI 中显式调用带 coverage 的 pytest 才会生效）。
- 静态检查：**Ruff**、**mypy**（见 `pyproject.toml`）；与测试互补，建议在提交前一并执行。

## 8. 本地缓存与 Git

- `.wikibench-cache/`、`.wikibench-results/` 等已在 `.gitignore` 中；diskcache 产生的 `cache.db` / `.cache.db` 亦应忽略，勿提交。
