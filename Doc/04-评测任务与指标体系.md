## 2. 质量任务详解

### 2.1 T1 · Retrieval Accuracy（检索准确性）

**回答的问题**：给定一个关于原始语料的事实性问题，编译后的 wiki 能否返回正确答案？

**输入**：`corpus.ground_truth.qa_pairs`

**执行流程**：
```
for qa in qa_pairs:
    response = adapter.query(qa.question)
    score = match(response.answer, qa.answer, qa.answer_type)
```

**打分算法**（按 `answer_type` 分派）：

| answer_type | 算法 | 备注 |
|------------|------|------|
| `short` | normalized exact match | 小写、去标点、去冠词 |
| `extractive` | token-level F1 | 对抗表述差异 |
| `numeric` | 数值相等 ± 容差 | 容差由 qa 指定 |
| `yes_no` | 二分类 accuracy | |
| `list` | set F1（基于归一化元素） | |
| `open_ended` | LLM-as-judge（可选） | 默认不启用，避免引入模型偏差 |

**核心指标**：
- `retrieval_accuracy` —— 全体 qa 的平均分
- `retrieval_accuracy@k` —— 取 top-k candidate answers（若 adapter 支持）
- `accuracy_by_difficulty` —— 按 easy/medium/hard 分层
- `accuracy_by_source_count` —— 按"答案散布在多少文档里"分层（检验跨文档整合能力）

---

### 2.2 T2 · Knowledge Fidelity（知识保真度）

**回答的问题**：编译后的 wiki 是否保留了原始语料中的关键事实？是否引入了原始语料中不存在的事实（幻觉）？

**输入**：`corpus.ground_truth.fidelity_claims`（含正样本 `should_exist=true` 与负样本 `should_exist=false`）

**执行流程**：
```
for claim in fidelity_claims:
    verdict = adapter.query(f"以下说法在知识库中是否有支持？{claim.text}")
    # verdict ∈ {supported, not_supported, unknown}
```

**核心指标**（把任务看成二分类）：

| 指标 | 定义 | 含义 |
|------|------|------|
| `fidelity_recall` | TP / (TP + FN) | 正样本被识别出的比例——关键事实保留率 |
| `hallucination_rate` | FP / (FP + TN) | 负样本被错误确认的比例——幻觉率 |
| `fidelity_f1` | F1(TP, FP, FN) | 综合指标 |
| `unknown_rate` | 回答 unknown 的比例 | 过度保守 vs 过度自信 |

> **关键设计**：`unknown` 是允许的第三个选项，避免"强制二选一"导致的虚假分数。`hallucination_rate` 只统计明确错的部分。

**打分实现选择**：
- **默认**：规则匹配 adapter 返回的结构化字段 + 关键词正则
- **可选**：LLM-as-judge（配置 `--judge-model`），用于 adapter 返回自由文本的情况

---

### 2.3 T3 · Contradiction Detection（矛盾检测）

**回答的问题**：原始语料中故意埋入的矛盾，wiki 能否识别并标注？

**输入**：`corpus.ground_truth.contradictions`

**执行流程**：
```
for c in contradictions:
    response = adapter.query(
        f"关于{c.topic}，知识库中是否存在相互矛盾的陈述？"
    )
    # 期望 response 包含：{has_contradiction: bool, conflicting_snippets: list}
```

**核心指标**：

| 指标 | 定义 |
|------|------|
| `contradiction_recall` | 召回率：真实矛盾被识别的比例 |
| `contradiction_precision` | 精度：识别为矛盾的里面有多少真矛盾 |
| `contradiction_f1` | F1 |
| `recall_by_type` | 按 numeric/temporal/entity/policy 分层 |

**与保真度的区别**：
- T2 问"单一事实是否保留"
- T3 问"多事实之间的一致性是否被监控"

---

### 2.4 T4 · Grounding（溯源可追踪性）· Phase 1.5

**回答的问题**：wiki 的回答是否能追溯到原始文档？

**打分**：
- 对 T1 的每个正确回答，检查 adapter 返回的 `sources` 字段
- `grounding_accuracy` = 实际来源在声明 sources 中的命中率

**为什么重要**：企业场景下"来源可查"比"答案正确"更重要。

---

### 2.5 T5 · Opinion Synthesis（观点综合）· Phase 1.5 · Forum corpus 专属

**这是 WikiBench 最独特的任务**，也是现有任何基准都没有的评测维度。

**回答的问题**：面对同一个话题下存在**多种对立观点**的论坛讨论，wiki 能否：
1. 准确识别"主流共识"是什么
2. 不丢失"有价值的少数派观点"
3. 不把"高噪声的错误意见"误当成共识
4. 识别"观点随时间的演化轨迹"

**前提条件**：corpus 含 `opinion_map.jsonl`（见 `03-数据集与语料库设计.md` 第 5.5 节）

**输入**：`corpus.ground_truth.opinion_map`

**执行流程**：

```python
for om in opinion_map:
    # 子任务 1：主流共识识别
    resp_consensus = adapter.query(
        f"关于"{om.topic}"，综合所有讨论，主流观点是什么？"
    )
    # 子任务 2：少数派观点
    resp_minority = adapter.query(
        f"关于"{om.topic}"，有哪些重要的反对意见或少数派观点？"
    )
    # 子任务 3：时间演化（如存在）
    if om.temporal_shift:
        resp_shift = adapter.query(
            f"关于"{om.topic}"，观点是否随时间发生了变化？"
        )
```

**打分算法**：

| 子任务 | 算法 | 说明 |
|--------|------|------|
| 主流共识识别 | LLM-as-judge（与 `ground_truth.main_consensus` 对比） | 语义相似度打分 [0,1] |
| 少数派覆盖 | Set recall（normalized）：预测集 ∩ 真实少数派集 / 真实集 | 防止丢失重要异见 |
| 幻觉观点率 | 预测中不在任何 `source_posts` 支持的观点比例 | 越低越好 |
| 时间演化识别 | 二分类（识别/未识别）× 描述准确度 | 仅含 temporal_shift 的样本 |
| **支持率校准** | \|predicted_ratio - actual_support_ratio\| | 不允许把 18% 的少数派说成"多数人认为" |

**核心指标**：

| 指标 | 定义 | 范围 | 越高越好？ |
|------|------|------|----------|
| `consensus_accuracy` | 主流共识识别的语义准确度 | [0,1] | ✅ |
| `minority_recall` | 重要少数派观点的覆盖率 | [0,1] | ✅ |
| `opinion_hallucination_rate` | 无来源支撑的幻觉观点比例 | [0,1] | ❌ |
| `ratio_calibration_error` | 支持率描述与实际的平均偏差 | [0,1] | ❌ |
| `temporal_shift_recall` | 时间演化识别率（含样本才计算） | [0,1] | ✅ |
| `opinion_synthesis_f1` | 综合 F1 | [0,1] | ✅ |

**为什么这个任务特别重要**：

一个 LLM Wiki 实现完全可以在 T1/T2/T3 上表现优秀（因为那些任务都基于结构化事实），但在 T5 上暴露根本性缺陷——它可能把"100 楼激烈争论"简化为一句话，丢失了所有有价值的分歧。T5 真正测试的是：**wiki 能否保留集体智慧，而不是单一权威声音**。

**T5 vs T3 的区别**：
- T3 测"文档内已知矛盾的识别"——矛盾是事先埋入的
- T5 测"论坛中自然涌现的多元观点综合"——没有预设答案，只有社区行为信号

## 3. 效率任务详解

### 3.1 E1 · Ingest Cost（编译成本）

**测量**：`adapter.ingest(docs)` 调用期间：
- `ingest_tokens_in` —— 送入 LLM 的总 token
- `ingest_tokens_out` —— LLM 产出的总 token
- `ingest_duration_s` —— wall clock
- `ingest_calls` —— LLM 调用次数
- `ingest_cost_usd` —— 估算成本（基于模型价目表）

**派生指标**：
- `ingest_cost_per_doc` = 总成本 / 文档数
- `ingest_cost_per_fact` = 总成本 / ground_truth 事实数

### 3.2 E2 · Query Efficiency（单次查询成本）

**测量**：所有 query 的 token 平均值
- `query_tokens_avg`
- `query_latency_p50`, `query_latency_p95`
- `query_cost_usd_avg`

### 3.3 E3 · Token Ratio（压缩比）

**核心公式**：
```
token_ratio = wiki_tokens / raw_corpus_tokens
```
- `wiki_tokens`：adapter 的编译产物 token 数（通过 `adapter.export_wiki()` 或记账）
- `raw_corpus_tokens`：原始 corpus 全部文档 token 数

**派生**：`breakeven_sessions = ingest_cost / (per_session_raw_tokens - per_session_wiki_tokens)`
—— 编译在多少次查询后回本。

> ⚠️ **陷阱**：`token_ratio` 越低不一定越好。必须与 `fidelity_recall` 联合看。极端压缩（= 0.01）但 fidelity 崩溃是"压坏了"。

## 4. 可维护性任务详解

### 4.1 M1 · Incremental Update（增量更新成本）

**前提**：adapter 实现了 `update(new_docs, removed_docs)` 方法

**流程**：
1. 初始 ingest corpus v1
2. 注入变更（新增 / 删除 / 修改 10% 文档）
3. 两种策略：
   - **incremental**：调用 `adapter.update()`
   - **rebuild**：丢弃旧 wiki，重新 `ingest(v2)`
4. 比较两种策略的：
   - 成本（token 数）
   - 结果质量（再跑一次 T1/T2/T3）

**核心指标**：
- `incremental_cost_ratio` = `incremental_cost / rebuild_cost`
- `incremental_quality_delta` = 增量更新后相对重编译的质量损失

### 4.2 M2 · Staleness Detection（过期识别）· Phase 1.5

**前提**：corpus 含 `staleness_events.jsonl`（某 fact 在某时间点被标记过期）

**流程**：
```
for event in staleness_events:
    # 模拟在 fact 过期后查询
    response = adapter.query(event.question, as_of=event.timestamp_after)
    # 期望 adapter 返回 "该信息可能已过期" 或 "无匹配"
```

**指标**：
- `staleness_precision` / `recall` / `f1`

### 4.3 M3 · Coverage Calibration · Phase 2

**前提**：adapter 返回回答时附带自评 `coverage: high/medium/low`

**核心指标**：
- 对比自评覆盖度 vs 实际 T1 accuracy
- `calibration_error` = |声称覆盖度 - 实际准确度|

## 5. 综合评分（Composite Score）

### 5.1 设计原则

- **不做黑盒加权**：所有权重公开，算法在 `metrics/aggregators/composite.py`
- **多套评分共存**：不同场景有不同 composite（"研究版" vs "企业版" vs "效率优先"）
- **拒绝单一榜单霸权**：Leaderboard 支持按单指标排序

### 5.2 默认 Composite（WikiBench Score v1）

```
WikiBench_Score = 0.45 × Quality + 0.25 × Efficiency + 0.30 × Maintainability

Quality       = mean(retrieval_accuracy, fidelity_f1, contradiction_f1)
Efficiency    = f_eff(token_ratio, ingest_cost_per_doc)    # 归一化后取反
Maintainability = mean(1 - incremental_cost_ratio, staleness_f1, calibration_score)
```

每个子项归一化到 [0, 1]，composite 最终也是 [0, 1]。

### 5.3 反 gaming 机制

- **Floor 约束**：任何 sub-metric < 0.5 时，该维度整体被降级（例：fidelity_recall < 0.5 → Quality 上限为 0.5）
- **对抗性任务必跑**（Phase 2）：公开 corpus 之外，榜单运行附加 hold-out 对抗性 corpus

## 6. LLM-as-Judge 使用规范

**原则**：尽量避免，能用规则就用规则。必须用时：

| 规则 | 说明 |
|------|------|
| 必须可替换 | 支持 `--judge-model openai:gpt-4o / anthropic:claude-3-5-sonnet / local` |
| 必须跑双判 | 两个不同模型独立打分，分歧时走人工 / 保守处理 |
| 必须记录 judge 版本 | 写进 `BenchmarkResult.environment` |
| 成本报告 | judge 调用的 token 单独列出 |

**不允许**：使用同一模型既做 adapter backend 又做 judge（自我评分偏差）。

## 7. 指标定义字典（摘录）

| 指标 ID | 公式 / 定义 | 范围 | 越高越好？ |
|--------|-----------|------|----------|
| `retrieval_accuracy` | Σ match(pred, gold) / N | [0,1] | ✅ |
| `fidelity_recall` | TP / (TP + FN) on positive claims | [0,1] | ✅ |
| `hallucination_rate` | FP / (FP + TN) on negative claims | [0,1] | ❌ 越低越好 |
| `contradiction_f1` | F1 of detected vs seeded | [0,1] | ✅ |
| `opinion_synthesis_f1` | 观点综合综合 F1（仅 forum corpus） | [0,1] | ✅ |
| `opinion_hallucination_rate` | 无来源幻觉观点比例（仅 forum corpus） | [0,1] | ❌ |
| `token_ratio` | wiki_tokens / raw_tokens | (0, ∞) | ❌ 越低越好，但受 fidelity 约束 |
| `ingest_cost_usd` | Σ API cost in ingest | [0, ∞) | ❌ 越低越好 |
| `incremental_cost_ratio` | incremental / rebuild | [0, 1] | ❌ 越低越好 |
| `staleness_f1` | F1 of staleness detection | [0,1] | ✅ |
| `calibration_error` | ‖claimed_coverage - actual_acc‖ | [0,1] | ❌ 越低越好 |

完整字典维护在 `wikibench/metrics/_registry.py`，对应文档自动生成。

## 8. 未决议题

- [ ] **Retrieval answer_type 是否需要"reasoning"类？** —— 倾向：Phase 1.5 引入，默认关闭
- [ ] **对 adapter 是否透出 LLM-as-judge？** —— 不透出，避免 adapter "伪装答案"
- [ ] **矛盾检测是否区分"识别"和"解决"？** —— 是，但 MVP 只测识别
- [ ] **T5 打分的 LLM-as-judge 偏差问题**：主流共识的语义评分依赖 judge 模型，可能引入文化/语言偏差
  - 缓解：提供多语言 judge prompt 模板 + 用人工标注集做 judge 校准
- [ ] **T5 是否只适用于 forum corpus？** 还是结构化文档也可以做"多作者观点综合"？
  - 倾向：扩展为通用任务，但 forum corpus 是主战场
- [ ] **Composite Score 是否需要按语料 tier 归一化？**（不同 tier 难度不同）
  - 倾向：是，每个 tier 独立 composite，不跨 tier 比较
- [ ] **T5 的 support_ratio 基准**：社区投票分布作为 ground truth 是否过于"民主暴力"？
  - 即：高票不一定对，低票不一定错（反例：历史上的少数正确派）
  - 倾向：support_ratio 仅作辅助指标，不是 T5 的主指标；主指标用专家标注
